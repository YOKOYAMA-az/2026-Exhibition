# 海外出張スケジュール表作成

海外出張の出発前に、社内承認用の「海外出張スケジュール表」（A4横置き・1枚）を作成するコマンド。展示会視察レポート（`make-report.md`系列）が**帰国後**の成果物であるのに対し、これは**出発前**の承認申請書である。

## 参照するファイル

1. `Images/海外出張スケジュール表フォーマット.jpeg`（社内の正式フォーマット。存在しない場合はユーザーに確認する）
2. 対象出張フォルダー内の一次情報（優先順）
   - `Itinerary_Budget.md`（旅程・予算がまとまっている場合）
   - `PreVisit_Briefing.md`（視察目的・背景の補足）
   - `Nippou.txt`／その他のメモ
3. ユーザーから直接提示される情報（出張者・出張先・航空券／ホテル見積り等）

## フォーマットの構成要素（参考画像に忠実に）

1. **表題行**：「海外出張スケジュール表」＋出張先／出張者＋承認欄（社長・部門長・部門長・作成者の4列、作成者列には氏名と作成日を入れる）
2. **メイン日程表**：列は「日／曜／国名／訪問先＆フライト／アポ・終日／ホテル／見積額」の7列。1日1行を基本とし、便名・出発/到着時刻・宿泊先・当日の見積額をその日の行にまとめる
3. **見積合計表**：出張者ごとに列を分ける（座席クラスが異なる場合は必ず分ける）。フライト代・ホテル代・小計・合計を行で持つ
4. **備考欄**：運賃条件・キャンセル規定・ビザ要件・未確定事項など
5. **目的／狙いの要約**：タイトル直下（参考画像では空欄になっている部分）に、出張の目的・狙いを2〜4行程度で要約して入れる
   - 情報源は`PreVisit_Briefing.md`の「視察の目的」章（あれば）。番号付き箇条書きを①②③…の形で1段落に圧縮する
   - この欄を確保するため、出張先／出張者のラベル・値の列幅は必要最小限（ラベル列2cm程度、値列4.5cm程度）に絞り、浮いた幅をすべて目的欄に回す。目的欄はタイトル列と合わせて13cm前後を目安にする
   - 出張先／出張者の値は1行に収まらなくてもよい（列を折り返す）。目的欄のほうを優先して広く取る

## 出力

- Word形式（`.docx`、A4横置き）：`Reports/<出張フォルダー>/海外出張スケジュール表_<出張先>.docx`
- 生成には `python-docx` を使う。システムPythonは外部管理のため、venvを作って導入する：
  ```bash
  python3 -m venv /tmp/scratch/venv && source /tmp/scratch/venv/bin/activate
  pip install python-docx --quiet
  ```
- Google Driveへの複製が求められた場合は「Google Drive連携時の制約」章に従う

## 実行手順

1. 参考フォーマット画像をReadツールで開き、レイアウト（列構成・承認欄・罫線）を確認する
2. 対象出張フォルダーの一次情報を読み、出張者・日程・便名・時刻・宿泊先・見積額を確認する。フォーマット画像と一次情報の間で日程や金額に食い違いがあれば、一次情報を優先しつつユーザーに確認する
3. `python-docx`でA4横（`Cm(29.7) x Cm(21.0)`、`WD_ORIENT.LANDSCAPE`）のドキュメントを組み立てる
   - 表題・承認欄テーブル → メイン日程テーブル → 見積合計テーブルの3段構成
   - 罫線は**セル単位で`tcBorders`を手組みしない**（結合セルの境界で点線化する不具合が実際に起きた）。各テーブルに`table.style = 'Table Grid'`を適用したうえで、`tblPr`に`tblBorders`（top/left/bottom/right/insideH/insideV、`val="single"`）を明示指定する。テーブルごとに1回呼べば済むヘルパー関数にしておく
4. **A4一枚に収める工夫**
   - 見積合計表は右寄せで幅を絞り、その左に生まれる余白へ備考欄を差し込む（1つのテーブル内で、左列に備考の複数行、右側に合計表の列を並べる構成。詳細は本ファイル末尾の実装メモを参照）
   - 各セルは**1〜2行に収める**ことを基本にする。特にメイン日程表の「ホテル」列は連泊が多いと縦に膨らみやすいので、次の項目のルールで圧縮する
   - **ホテル欄の連泊表記**：ホテル名・フルネーム・（朝食込み等の付帯情報）は**チェックイン日の行にのみ**記載する。2日目以降の連泊日は「〃（連泊）」、チェックアウト日は「〃（チェックアウト）」とだけ書き、宿泊費もチェックイン日の行に集約する（各日の行で名称を繰り返さない）
   - 見積額・備考など長くなりがちな列は、ラベルと金額を「／」区切りで1行にまとめるなど、行数を削ってでも1〜2行に収める。要確認事項が長文になる場合は日程表のセルに書かず、末尾の備考欄に移す
5. 生成した`.docx`を対象フォルダーに保存する
6. 未確定事項（延泊要否・運賃確定額等）は文中に「［要確認：〇〇］」として残す。推測で埋めない

## Google Drive連携時の制約（実施経験に基づく重要な注意）

Google Driveへ「ドキュメント形式（Google Docs）」で複製してほしいと言われた場合、以下を踏まえて対応する。

- **`.docx`のバイナリをbase64化して`create_file`にそのまま渡す方法は使わない。** ファイルサイズが数十KBでも、base64を読み取ってから手で書き戻す過程で内容が欠落・破損するリスクが高く、実際に途中で切れて別の文字列に化けたことがある。壊れたファイルをアップロードするくらいなら、この経路は最初から採らない
- 確実なのは **HTML（`textContent` + `contentMimeType: "text/html"`）をGoogle Driveの`create_file`に渡し、Google Docsへ自動変換させる方法**
- ただしこの変換には2つの既知の制約がある
  1. **`rowspan`/`colspan`によるセル結合は結合されず、span数だけ内容が複製される。** 備考欄を5行分の`rowspan`にしたところ、同じ文章が5回並んでしまった。→ **結合は一切使わず、単純な1行1データの表に分解する**（合計と備考は横並びにせず、合計表→備考の順で縦に並べる）
  2. **`<td>`内に入れ子にした`<table>`は列がずれて崩れる。** 備考と合計表を横並びにしようとして入れ子テーブルにしたところ、外側の表の列数に引きずられて中の行がずれた。→ **入れ子テーブルも使わない**
  3. **HTMLの`@page`によるページの向き（A4横置き）はGoogle Docsに引き継がれない。** アップロード後は必ず縦置きになる。ユーザーに「ファイル→ページ設定→用紙の向き→横」を手動で設定してもらうよう案内する
- まとめると、Google Drive用のHTMLは「結合なし・入れ子なしの単純な表を縦に並べるだけ」に設計し、ページの向きは手動設定を案内する。**Word版（`.docx`）が正式なA4横1枚のマスターであり、Google Docs版は内容確認用の複製という位置づけ**であることをユーザーに伝える
- アップロード後は`get_file_metadata`で内容（`contentSnippet`）を必ず確認し、結合による重複や列ずれが起きていないか検証してから完了報告する

## 実装メモ：備考欄と合計表を1枚に収める配置（Word版）

`python-docx`で以下の構成にすると、見積合計表の左に生まれる余白へ備考欄を差し込める。

```python
footer_table = doc.add_table(rows=5, cols=4)
# col0: 幅広（備考用）、col1-3: 合計表（ラベル／出張者A／出張者B）
# col0 は rows 0-4 を merge_cells で縦結合し、備考の複数行を1セルに収める
# col1-3 は結合せず、行ごとに「項目名／出張者A金額／出張者B金額」を入れる
# 最終行（2名合計）は col2-3 をmerge_cellsで結合して1つの金額を表示する
```

Word（docx）では`merge_cells`によるセル結合が正しく描画される。Google Drive変換時の制約（結合が複製される）とは別問題なので混同しないこと。

## 不明な点の扱い

- 航空券・ホテルの見積額は、確認した時点の予約画面表示額であることを明記し、「予約確定後に確定額へ差し替え」と注記する
- 出張者ごとに座席クラス・見積額が異なる場合は必ず両者を併記する。片方だけ書いて他方を省略しない
- 延泊要否・部屋割り等、現地手配が未確定の項目は「［要確認：〇〇］」と明記し、憶測で埋めない

## 既存ファイルを上書きする場合のルール

- 同名の`.docx`が既に存在する場合は上書きせず、ユーザーに再生成の意図を確認してから置き換える
- Google Drive側に作成済みのファイルがある場合、Drive操作ツールにファイル更新（上書き）機能がないため、内容を直す場合は新規ファイルとして作り直す。同名の古いファイルが残る場合はユーザーに手動削除を依頼する（誤って別のファイルを削除しないよう、削除は必ずユーザー自身に行ってもらう）
